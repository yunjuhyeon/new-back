✔ 타임리프 템플릿 반환 과정에서 fetch 문제를 해결하려니 경로 설정과 데이터 매핑에서 막막한 점이 많았다. 처음엔 템플릿 파일이 없어서 발생한 문제인지, 데이터 전달 과정에서 오류가 있는지조차 명확히 알 수 없었고, 검색을 통해 해결하려 해도 직접 코드와 설정을 확인하며 하나씩 점검하는 수밖에 없었다.

문제의 원인이 단순히 템플릿 경로나 데이터 전달 누락이라는 기본적인 사항임을 알고나서 허무했지만, 설정을 정확히 이해하고 직접 수정하여 문제를 해결했다는 점에서 큰 뿌듯함을 느꼈다. 새로운 기술을 다룰 때 기본에 충실한 접근이 얼마나 중요한지를 다시금 배우는 계기가 되었다.

또한, 백엔드에서 페이지네이션 정보를 포함한 응답 구조를 설계해 동적으로 페이지 이동이 가능하도록 설정

<h3>6-2 총평</h3>
<strong>✨설계:초반 기획부터 세부적으로 완성하자 !</strong><br><br>
이번 프로젝트를 진행하면서, 기획과 설계 단계에서 사소한 부분까지 왜 고려해야하는지 깨달았습니다. 
처음에는 전반적인 기능 흐름만 생각하며 작업을 시작했지만, 실제 구현 단계에서 예상치 못한 문제들이 자주 발생했습니다.
<br>
<br>
특히, <strong>고객센터 문의 작성 및 조회 기능</strong>을 구현하려 했을 때, 화면 설계에서 삭제버튼 영역, 필터링 영역등이 제대로 구현되어 있지 않아서, 
원하는 기능을 제때 제대로 구현하지 못한 불편함이 있었습니다.<br>
이 경험을 통해, <strong>화면 설계가 서버 로직과 긴밀히 연계되어야 하며, 초기 단계에서 이를 충분히 고려하는 것이 중요하다는 것</strong>을 배웠습니다.
<br>
앞으로는 설계 단계에서 사용자 경험과 세부 흐름을 더욱 꼼꼼히 준비해 문제를 최소화하고자 합니다.
<br>
<br>
<strong>✨같이의 가치:프로젝트는 혼자하는 것이 아니다 !</strong><br><br>
이번 프로젝트를 진행하면서 깨달은 가장 중요한 점은, 나의 작업뿐만 아니라 팀원들의 작업이 얼마나 중요한지에 대한 것입니다. 
제가 맡았던 <strong>관리자페이지</strong>는 다른 팀원들이 맡은 <strong>게시글, 게시글의 신고하기</strong>와 밀접하게 연결되어 있었습니다.

제가 맡은 기능이 아무리 완벽하게 구현되었더라도, 팀원들의 작업이 완료되지 않으면 제 작업은 의미가 없어지거나 중단될 수밖에 없었습니다.
이 과정에서 팀원들의 작업 진행 상황에 따라 기획과 설계를 수정해야 했고, 이러한 협업을 통해 각자의 작업이 상호 의존적임을 느꼈습니다.

또한, <strong>메서드 이름이나 DTO 구조</strong>가 사전에 명확히 정의되지 않아 협업 중에 문제가 발생하기도 했습니다. 
예를 들어, 다른 서비스에서 사용하는 DTO 이름이 변경되었지만 그 변경 사항이 공유되지 않아, <strong>관리자 페이지</strong>에서 게시글 데이터를 불러오는 기능이 제대로 작동하지 않는 문제가 있었습니다.
이후, 코드 변경 사항은 반드시 공유하고, 수정된 부분이 다른 작업에 미칠 영향을 사전에 점검하는 습관을 들이게 되었습니다.

이와 더불어,<strong>Git 충돌</strong> 문제도 협업 과정에서 종종 발생했습니다.
같은 파일을 여러 명이 동시에 수정하거나 작업 내용이 충분히 공유되지 않아 이미 해결된 문제를 다시 수정하는 비효율적인 상황이 생기곤 했습니다.
이러한 경험을 통해, 협업은 단순히 자신의 작업을 잘 수행하는 것이 아니라, <strong>팀원들과 작업 상태를 명확히 공유하고 소통하는 것</strong>이라는 점을 깨닫게 되었습니다.

이번 프로젝트를 통해, <strong>'내 작업은 팀에 영향을 미친다'</strong>는 점을 깊이 인지하게 되었고, 앞으로는 더 적극적으로 소통하며,
협업 과정에서의 문제를 최소화하기 위해 기획 단계부터 명확한 규칙과 협의 사항을 설정하고 이를 실천해 나가고자 합니다.
<br><br>
<strong>✨미래: 새로운 도전을 두려워하지 않는 개발자!</strong><br><br>
이번 프로젝트를 시작하며 세운 목표는 '새로운 기술에 도전하고 성장하기'였습니다.
저는 기존에 경험해보지 못했던 기술에 도전하고 싶었고, 프로젝트 초기 파트 배분 단계에서 <strong>fetch를 활용한 필터링, 검색, 데이터 삭제 및 조회 기능</strong>과 같은 작업을 자원해 맡게 되었습니다.

특히, <strong>JavaScript fetch API</strong>를 사용하여 클라이언트와 서버 간 데이터를 주고받는 비동기 로직을 구현하며, 사용자 중심의 기능을 효율적으로 구현하는 방법을 이해하게 되었습니다. 
필터 조건과 검색 키워드에 따라 서버에 요청을 보내고, 응답 데이터를 화면에 동적으로 렌더링하는 과정은 처음이라 시행착오가 많았습니다.
검색 조건에 따라 데이터를 제대로 불러오지 못하거나, 삭제 요청을 처리하는 과정에서 상태가 즉시 반영되지 않는 문제가 발생하기도 했습니다.

하지만 이러한 문제를 하나씩 해결하면서, <strong>서버와 클라이언트 간 데이터 흐름을 효율적으로 설계하고 처리하는 방법</strong>을 익힐 수 있었습니다.
<strong>fetch</strong>를 활용해 데이터를 가져오고, 페이지 이동이나 상태 동기화를 통해 사용자 경험을 향상시키는 방법을 배우며 기술적 성장을 이룰 수 있었습니다.

이 과정에서 단순히 새로운 기술을 배우는 것을 넘어, <strong>유지보수성과 확장성을 고려한 설계</strong>가 얼마나 중요한지 깨달았습니다.
